# FCF — Принципы работы

## Единая Вычислительная Архитектура (ЕВА)

Полное техническое описание принципов, математического фундамента и механизмов работы системы.

---

## 1. Основополагающая идея

ЕВА отвергает парадигму «предобученная модель + файнтюнинг». Вместо этого система начинает существование с **чистого листа** — единственного слоя трансформера, все матрицы которого инициализированы случайным образом (равномерное распределение Xavier/Glorot). Никакие внешние веса не импортируются.

Система не отделяет «память» от «вычислений». Единственная форма существования информации — **матричное состояние**: либо явный тензор `(Key, Value)`, либо компактный латентный код `z ∈ ℝᵈ`. Знание о мире не хранится как факт или правило — оно представляет собой **способность воспроизвести определённое поведение** при активации соответствующих матричных преобразований.

Рост архитектуры не предопределён. Система расширяется только тогда, когда существующая структура объективно не справляется с задачей, что диагностируется внутренними метриками (SRG, градиентные нормы, история уверенности).

## 2. Математический фундамент

### 2.1. Атомарный алфавит (SVD-базис)

Каждая обучаемая матрица слоя `W` может быть факторизована через сингулярное разложение:

```
W = U Σ Vᵀ = Σᵢ σᵢ · uᵢ · vᵢᵀ
```

Первые K компонент (например, K=256) сохраняются как **атомы**. Любая близкая матрица ΔW может быть выражена как линейная комбинация атомов:

```
ΔW = Σᵢ cᵢ · σᵢ · uᵢ · vᵢᵀ
```

Это позволяет компактно представлять модификации весов и оперировать ими в латентном пространстве коэффициентов cᵢ, а не в полном пространстве параметров.

### 2.2. Факторизованное латентное состояние

Для компактности состояние кодируется латентным кодом `z ∈ ℝᵏ` (k ≪ d). Гиперсеть генерирует z из контекстного вектора, а слое-специфичные декодеры разворачивают его в низкоранговые матрицы A, B:

```
ΔW ≈ A(z) · B(z)ᵀ
```

где A ∈ ℝᵈˣʳ, B ∈ ℝᵈˣʳ. Ранг r распределён иерархически: r=4 для нижних слоёв (базовая грамматика), r=8 для средних (доменная информация), r=16 для верхних (сложные рассуждения).

### 2.3. Архитектура трансформера

Базовый вычислительный блок — стандартный декодерный слой с причинным вниманием:

**Multi-Head Causal Self-Attention:**

```
Q = X · W_Q,  K = X · W_K,  V = X · W_V

Attention(Q, K, V) = softmax(QKᵀ/√dₖ + causal_mask) · V

MultiHead = Concat(head₁, ..., headₕ) · W_O
```

Позиционное кодирование — Rotary Position Embedding (RoPE), применяемое к Q и K до вычисления attention scores:

```
freqs = θ⁻²ⁱ/ᵈ,  где θ = 10000
Q_rope = rotate(Q, freqs),  K_rope = rotate(K, freqs)
```

**SwiGLU Feed-Forward Network:**

```
FFN(x) = (SiLU(x · W_gate) ⊙ (x · W_up)) · W_down
```

Активация SiLU (Sigmoid Linear Unit) с гейтовым механизмом обеспечивает нелинейность с адаптивным пропусканием сигнала.

**Pre-Norm архитектура с RMSNorm:**

```
y = x + Attention(RMSNorm(x))
y = y + FFN(RMSNorm(y))
```

RMSNorm стабилизирует обучение без батч-нормализации, нормируя по RMS вместо дисперсии.

### 2.4. Контекстный вектор

Контекстный вектор c_query вычисляется как скрытое состояние последнего токена после прямого прохода через слой. Этот вектор служит компактным представлением семантического содержания всей входной последовательности и используется для:

- Поиска похожих состояний в FAISS-хранилище
- Вычисления семантического сходства в SRG
- Маршрутизации запросов к доменным адаптерам
- Инициализации латентного кода z для KCA-цикла

## 3. Semantic Relevance Gate (SRG)

SRG — механизм самооценки, вычисляющий уверенность системы в сгенерированном ответе. Итоговая оценка складывается из трёх взвешенных компонент:

```
confidence = w_sim · S(c_query, c_response)
           + w_ent · H_entropy
           + w_eth · E_ethics
```

### 3.1. Семантическое сходство

```
S(c_query, c_response) = (cos(c_query, c_response) + 1) / 2
```

Косинусное сходство между контекстными векторами запроса и ответа, нормализованное в диапазон [0, 1].

### 3.2. Энтропийная уверенность

```
H_entropy = 1 - H(p) / log₂(V)
```

где H(p) — энтропия распределения логитов последнего токена, V — размер словаря. Высокая энтропия означает неуверенность модели в выборе токена.

### 3.3. Этический скор

Вычисляется EthicsFilter (см. раздел 5). Если `E_ethics < θ_ethics`, ответ безусловно отклоняется (confidence = 0).

### 3.4. Режимы работы

На основе confidence SRG выбирает один из трёх режимов:

| Режим | Условие | Действие |
|-------|---------|----------|
| **Direct** | confidence ≥ 0.7 | Ответ возвращается сразу |
| **Reasoning** | 0.3 ≤ confidence < 0.7 | Запуск KCA-цикла или рекурсии |
| **Variational** | confidence < 0.3 | Ответ с пометкой о неуверенности |

## 4. FAISS State Storage

Векторная база данных на базе FAISS `IndexFlatIP` (inner product ≈ cosine similarity после L2-нормировки). Каждый «слепок» (snapshot) содержит:

| Поле | Тип | Описание |
|------|-----|----------|
| c | np.ndarray (d_model,) | L2-нормированный контекстный вектор (индексируется в FAISS) |
| K | np.ndarray (num_heads, head_dim) | Key-тензор последнего токена |
| V | np.ndarray (num_heads, head_dim) | Value-тензор последнего токена |
| confidence | float | Оценка SRG на момент сохранения |
| domain | str | Идентификатор домена |
| timestamp | float | Время создания (Unix) |
| usage_count | int | Счётчик использований |

**Алгоритм поиска:**
1. Контекстный вектор запроса L2-нормируется
2. FAISS выполняет поиск ближайшего соседа (inner product)
3. Если расстояние ≥ threshold (0.7) — состояние найдено
4. Счётчик usage_count инкрементируется

**Управление размером:**
- Максимум 10 000 слепков на слой
- При переполнении удаляется самый старый (min timestamp)
- Sleep Mode выполняет кластеризацию и удаление устаревших

## 5. Этическое ядро (EthicsFilter)

Пять неизменяемых аксиом, вшитых в механизм принятия решений:

### Аксиомы

1. **Не навреди** — ответ не должен содержать призывов к насилию, дискриминации, незаконным действиям
2. **Будь честен** — если система не уверена, она обязана явно указать это
3. **Уважай приватность** — система не идентифицирует и не сохраняет личные данные
4. **Стремись к истине** — при обнаружении противоречия система должна пытаться его разрешить
5. **Будь полезен** — ответ должен быть направлен на помощь пользователю

### Реализация

Фильтр содержит четыре категории паттернов:

| Категория | Аксиомы | Примеры паттернов |
|-----------|---------|-------------------|
| HARM_PATTERNS | 1 | Насилие, террор, дискриминация, наркотики |
| PRIVACY_PATTERNS | 3 | Номера карт, телефоны, паспорта, email |
| DISHONESTY_PATTERNS | 2, 4 | Уверенность + одновременная неуверенность |
| USELESS_PATTERNS | 5 | Пустые ответы, отказы без объяснений |

Каждое совпадение снижает скор аксиомы на `PENALTY_PER_MATCH = 0.2`. Итоговый этический скор — среднее по всем аксиомам.

```
ethics_score = mean(scores.values())
if ethics_score < θ_ethics: reject  # безусловное отклонение
```

**Ключевое свойство:** EthicsFilter неизменяем. Ни обучение, ни рост архитектуры не могут модифицировать этические правила.

## 6. Curiosity Loop

Механизм автономной генерации уточняющих вопросов при систематической неуверенности.

### Алгоритм

1. Счётчик `curiosity_counter` инкрементируется при `confidence < 0.6`
2. При `confidence ≥ 0.6` счётчик сбрасывается
3. При достижении порога `CURIOSITY_THRESHOLD = 10`:
   - Формируется специальный промпт: система получает исходный запрос, неуверенный ответ и инструкцию «Сформулируй один чёткий уточняющий вопрос»
   - Выполняется авторегрессионная генерация через тот же слой (`max_new_tokens=64`)
   - Сгенерированный вопрос передаётся пользователю или сохраняется в `pending_clarifications`
4. Счётчик сбрасывается

**Ключевое свойство:** никаких жёстко заданных строк. Модель использует свои обучаемые матрицы для формулирования релевантного вопроса.

## 7. Growth Controller

Принимает решение о необходимости расширения архитектуры на основе метрик слоя.

### Сигналы

| Сигнал | Условие | Действие |
|--------|---------|----------|
| **EXPAND_WIDTH** | avg_confidence < width_threshold (0.5) И gradient_norm > gradient_threshold (1.0) | Создать LoRA-адаптер для нового домена |
| **EXPAND_DEPTH** | avg_confidence < depth_threshold (0.3) И patience исчерпана (20) И рекурсия не помогла | Кристаллизовать новый слой |
| **TRY_RECURSION** | avg_confidence < depth_threshold (0.3) И patience исчерпана | Попробовать рекурсивную обработку |
| **NO_GROWTH** | Все условия выше не выполнены | Архитектура справляется |

### Защита от взрывного роста

- **MAX_LAYERS = 100** — абсолютный лимит слоёв
- **MIN_LAYER_INTERVAL = 300 сек** — минимальный интервал между созданием слоёв
- **Сигналы накапливаются** — требуется 2-3 последовательных сигнала EXPAND_WIDTH/DEPTH

## 8. LoRA-адаптеры и Domain Registry

### LoRA (Low-Rank Adaptation)

Каждый адаптер представляет пару низкоранговых матриц `A (rank × in_dim)` и `B (out_dim × rank)`:

```
ΔW = B · A
W_effective = W + α · ΔW / rank
```

Адаптеры применяются к 7 целевым модулям трансформера:
- **Attention:** W_Q, W_K, W_V, W_O — все имеют размерность (d_model, d_model)
- **FFN gate:** gate_proj — (d_model, d_model×4)
- **FFN up:** up_proj — (d_model, d_model×4)
- **FFN down:** down_proj — (d_model×4, d_model)

При rank=8 адаптер добавляет 286 720 параметров (0.12% от базовых 233.5M).

### Domain Registry

Реестр хранит доменные правила (DomainRule):

```
DomainRule:
  domain_id: str           # идентификатор домена
  adapter_path: str        # путь к файлу .lora
  context_centroid: ndarray # центроид контекстных векторов домена
  usage_count: int         # счётчик использований
  confidence_history: list # история SRG-оценок
```

**Алгоритм маршрутизации запроса:**
1. Вычислить c_query (контекстный вектор запроса)
2. Поиск точного совпадения в StateStorage (threshold=0.95)
3. Если не найден — поиск домена по косинусному сходству с центроидами всех доменов (threshold=0.7)
4. Если домен найден — загрузить и применить адаптер к attention + FFN
5. Выполнить генерацию с адаптированными весами
6. Удалить адаптер, восстановить исходные веса
7. Обновить центроид домена (EMA с weight=0.1)

## 9. Рост в глубину

### Рекурсивная обработка

Перед созданием нового слоя система пробует «додумать» ответ многократным проходом:

```
x₀ = Transformer(x_input)
for i in 1..MAX_DEPTH:
    x_i = Transformer(x_{i-1})
    ответ = Generate(x_i)
    if SRG(ответ) > 0.7: return ответ  # успех
# После исчерпания лимита → сигнал к кристаллизации
```

Адаптивное демпфирование (ρ=0.85) предотвращает расходимость:
```
x_i = ρ · x_{i-1} + (1-ρ) · Transformer(x_{i-1})
```

### Кристаллизация нового слоя

1. Создать PrimordialLayer как глубокую копию последнего слоя (state_dict)
2. Инициализировать пустые хранилища (StateStorage, DomainRegistry)
3. Специализировать новый слой на `failed_queries` (100 шагов AdamW, lr=1e-5)
4. Добавить слой в sequential pipeline

**Sequential pipeline:** каждый слой обрабатывает вход последовательно:
```
x → Layer₀ → x₁ → Layer₁ → x₂ → ... → Layerₙ → output
```

## 10. Сон (Sleep Mode)

Фоновый процесс оптимизации архитектуры в периоды простоя:

### 10.1. Кластеризация слепков

- Извлечение всех векторов из FAISS
- Кластеризация алгоритмом HDBSCAN (min_cluster_size=5, metric=euclidean)
- Для каждого кластера: вычисление центроида, пометка избыточных слепков как кандидатов на удаление
- Fallback: KMeans если HDBSCAN недоступен

### 10.2. Удаление устаревшего

```
score = usage_count · exp(-λ · age / 86400)
```

- λ = 0.1 (затухание в днях)
- Слепки с нулевым usage_count и возрастом > TTL_IDLE_DAYS (7) удаляются
- Слепки с score < 0.01 удаляются

### 10.3. Дистилляция LoRA в базовые веса

Для доменов с usage_count ≥ 100 и средней уверенностью ≥ 0.8:
1. Извлечь ΔW из LoRA-адаптера
2. Прибавить ΔW к базовым весам всех слоёв: `W = W + ΔW`
3. Удалить адаптер из DomainRegistry

### 10.4. Слияние избыточных слоёв

Для каждой пары соседних слоёв:
1. Пропустить тестовый тензор через оба слоя
2. Вычислить MSE между выходами
3. Если MSE < 0.01 — усреднить матрицы и удалить второй слой

### 10.5. Дефрагментация FAISS

После массовых удалений FAISS-индекс перестраивается для восстановления оптимальной производительности.

## 11. KCA Engine (Knowledge-Conscious Attention)

Механизм итеративного уточнения латентного кода с гарантированной сходимостью.

### 11.1. Утилитарная функция

```
ℒ_KCA(z) = -λ₁ · SRG_conf(c_out, p)
          + λ₂ · D_KL(p || p_target)
          + λ₃ · ||c_out - c_target||²
```

где:
- `SRG_conf` — уверенность Semantic Relevance Gate
- `p` — распределение логитов модели
- `p_target` — эталонное распределение (центроид кластера)
- `c_out`, `c_target` — выходной и целевой контекстные векторы

### 11.2. Градиентный спуск по латентному коду

```
z_{t+1} = z_t - η_t · ∇_z ℒ_KCA(z_t)
```

где `η_t = η₀ · ρᵗ` (ρ=0.85) — адаптивное демпфирование.

Веса базовой модели заморожены — оптимизируется только латентный код z.

### 11.3. Протокол динамической сходимости

Гарантирует завершение цикла ≤ 5 итераций:

| Механизм | Условие | Действие |
|----------|---------|----------|
| **Адаптивное демпфирование** | Всегда | η_t = η₀ · ρᵗ |
| **Детектор осцилляции** | cos(∇_t, ∇_{t-1}) < -0.5 | Усреднение последних 3 состояний |
| **Монитор гейта** | γ < 0.05 дважды подряд | Досрочное завершение (SATURATED) |
| **Жёсткий лимит** | iteration ≥ 5 | Форсированная остановка |

## 12. State Algebra

Операторы композиции знаний в латентном пространстве без экспоненциального роста памяти.

### Пространство латентных кодов

Каждое матричное состояние представлено латентным кодом `z ∈ ℝᵏ`. Успешные коды образуют многообразие `ℳ_z`.

### Операторы

| Оператор | Формула | Применение |
|----------|---------|------------|
| **Суммирование** | z_{A+B} = Proj(z_A + z_B) | Аддитивные контексты |
| **Масштабирование** | z_{αA} = Proj(α · z_A) | Усиление/ослабление аспекта |
| **Вычитание** | z_{A\B} = Proj(clamp(z_A - β·z_B, -τ, τ)) | Фильтрация аспектов |
| **Кросс-аттеншн** | z_comp = Proj(CrossAttn([z_A; z_B])) | Сложные нелинейные взаимодействия |

### Проектор Proj_ℳ

Лёгкий автоэнкодер (d_model → bottleneck=256 → d_model), обученный на истории успешных латентных кодов. Проецирует произвольный вектор на многообразие валидных состояний, предотвращая выход за пределы изученного пространства.

## 13. Иерархический HNSW

Трёхуровневый индекс для эффективного поиска состояний:

| Уровень | Содержимое | Использование |
|---------|-----------|---------------|
| **0 (глобальный)** | Центроиды всех доменов | Маршрутизация запроса к домену |
| **1 (доменный)** | Контекстные векторы всех слепков домена | Поиск ближайшего состояния |
| **2 (слой-специфичный)** | Слепки по диапазонам слоёв (1-8, 9-16, ...) | Уточнение глубины обработки |

**Алгоритм поиска:**
1. Уровень 0 → косинусное сходство c_query с центроидами доменов → выбор домена
2. Уровень 1 → поиск ближайшего слепка внутри домена
3. Уровень 2 (опционально) → уточнение по глубине слоя

Обеспечивает логарифмическую сложность поиска при миллионах сохранённых состояний.

## 14. Когнитивный цикл

Полный цикл обработки запроса объединяет все компоненты системы:

### Фаза 1: Восприятие и поиск

```
1. Токенизация запроса + RoPE
2. Вычисление контекстного вектора c_query
3. Поиск в иерархическом HNSW:
   a. Уровень 0 → ближайший домен
   b. Уровень 1 → ближайшее состояние
4. При совпадении > 0.95 → использовать сохранённые K, V (Фаза 3)
5. При частичном совпадении (0.7–0.95) → инициализировать z интерполяцией → Фаза 2
6. Без совпадения → холодный старт: z₀ = случайный или через гиперсеть → Фаза 2
```

### Фаза 2: Порождение и коррекция (KCA)

```
1. Развернуть z_t в матрицы ΔW (LoRA-декодер)
2. Прямой проход LLM с модифицированными весами
3. Генерация ответа (авторегрессия)
4. Вычисление ℒ_KCA(z_t)
5. Градиент ∇_z ℒ → обновление z_{t+1}
6. Протокол сходимости → CONTINUE или остановка
7. При необходимости — создание доменного правила или нового слоя
```

### Фаза 3: Исполнение и сохранение


## 15. Резюме

FCF (ЕВА) — это самозарождающаяся когнитивная архитектура, реализующая следующие ключевые инновации:

1. **Tabula Rasa** — отказ от предобученных моделей, старт со случайных весов
2. **Compute-as-Memory** — информация как матричное состояние, без разделения памяти и вычислений
3. **Этическое ядро** — неизменяемые аксиомы, вшитые в механизм принятия решений
4. **Органический рост** — расширение архитектуры только по объективной необходимости
5. **Самооценка** — SRG как внутренний механизм контроля качества
6. **Консолидация** — Sleep Mode для поддержания архитектуры в оптимальном состоянии
7. **State Algebra** — композиция знаний без экспоненциального роста памяти
8. **KCA** — итеративное уточнение с гарантированной сходимостью
9. **Иерархический поиск** — трёхуровневый HNSW для эффективной маршрутизации

Система спроектирована как изолированный, полностью автономный проект, независимый от EVA-Ai.
